# DIVA Player プロジェクト整備 実装プラン（レビュー修正版）

更新日: 2026-07-18  
対象: `diva-player` と、関連する `diva-data-pipeline`  
目的: 現行機能を壊さずに文書の不整合と技術的負債を整理し、未解決不具合を検証可能な単位で解消する。

## 0. レビューで修正した重要点

元プランには、実行前に直すべき問題があったため、次のように変更する。

1. `diva-player/docs/` を「方針違反」と断定して一括削除しない。
   - Git履歴には、これらの文書を意図的に統合・復元・更新した直近コミットがある。
   - `diva-player/README.md` は現在これらを正式な参照先としている。
   - private な別リポジトリだけを唯一の文書入口にすると、`diva-player` 単独の利用者が参照できない。
   - 先に「公開リポジトリに残す情報」と「private 運用文書」の境界を決め、内容を対応表で確認してから統合またはアーカイブする。
2. `implementation_plan.md` は即時削除しない。
   - 完了項目、未完了項目、設計判断を確認し、必要な情報の移管後に `docs/archive/` へ移すか削除する。
3. 2つのリポジトリの変更を1コミットとして扱わない。
   - リポジトリごとにブランチ、差分確認、テスト、コミットを分離する。
4. リモートブランチ削除を最初の作業にしない。
   - 両ブランチに `main` 固有の追加コミットがあり、対象ブランチ固有のコミットはないことを確認済みだが、削除は機能整備と無関係で復旧性も下げる。
   - ローカルブランチ削除は任意、リモート削除は整備完了後に別作業として実施する。
5. DBテーブル削除は「HIGHの清掃」ではなく、独立した破壊的変更として扱う。
   - 静的検索では `play_sessions` と `session_plays` のコード参照は見つからなかったが、本番データ、外部クライアント、手動運用の利用有無は未確認である。
   - 読み取り調査、バックアップ、復旧手順、ステージング検証を通過するまでDROP migrationを作成・適用しない。
6. テスト数を実測値に更新する。
   - 2026-07-18時点で `25` test files、`95` tests が成功。lintとbuildも成功している。
7. `ARCHIVE/pdf_out.txt` は内容確認前提の削除候補とする。
   - Git管理中のOCR出力であり、DIVA Playerとは無関係に見えるが、削除は `diva-data-pipeline` 側の独立コミットで行う。

## 2. 実行方針と安全条件

- 既存の未コミット変更は上書き、破棄、混在させない。
- ドキュメント変更とコード変更を同じコミットに混ぜない。
- 破壊的DB変更、SBC反映、リモートブランチ削除は個別の承認ゲートを設ける。
- 「実装済み」の判定は、文書だけでなくコード、テスト、必要に応じて稼働環境で確認する。
- B4とB5は保留指示を維持し、明示的な再開指示なしに変更しない。

## 3. フェーズA: 変更保護と棚卸し

### A1. 作業開始前の記録

各リポジトリで次を保存・確認する。

```powershell
git status --short --branch
git log --oneline --decorate -15
git diff --stat
git diff
```

`diva-data-pipeline` の2ファイルの変更は、次のいずれかに分類する。

- 完了済み実装の記録
- 未実装の計画変更
- 単なる書式修正
- 今回の整備と無関係なユーザー変更

最後の分類は今回のコミットに含めない。

### A2. 文書対応表の作成

ファイル単位ではなく、以下の情報単位で `diva-player/docs` と `diva-data-pipeline/docs/diva-player/ACTIVE` を比較する。

| 情報 | 公開側に必要 | private側に必要 | 正本候補 |
|---|---:|---:|---|
| ローカル起動・テスト | はい | 任意 | `diva-player` |
| システム構成・API概要 | はい | はい | 公開可能部分は `diva-player` |
| SBCの接続先・秘密情報・復旧手順 | いいえ | はい | `diva-data-pipeline` |
| 現行機能と既知の制限 | はい | はい | 公開版と運用版を役割分担 |
| パイプライン運用 | 参照のみ | はい | `diva-data-pipeline` |
| 完了済み計画の詳細ログ | 原則不要 | archive | `ARCHIVE` |

受け入れ条件:

- 各情報に正本が1つ定義される。
- 公開READMEからprivateリポジトリだけに依存するリンクを必須導線にしない。
- 重複文書を残す場合は、役割と更新責任を冒頭に明記する。

## 4. フェーズB: ドキュメント整理

### B1. `diva-player` 側

推奨案は、`docs/` 全削除ではなく最小限の公開文書を残すこととする。

- `README.md`: セットアップ、主要コマンド、文書入口を維持。
- `docs/PROJECT_MANUAL.md`: 公開可能な構成・動作仕様を維持または `README.md` に統合。
- `docs/GUIDELINE.md`: 実際に適用する開発規約だけを維持。
- `docs/CURRENT_STATUS.md`: 公開してよい既知問題と検証日を簡潔に維持。
- `docs/SBC_OPERATIONS.md`: 秘密情報を含まない概要だけ残すか、private側への案内に縮小。
- `docs/WEB_STATUS_AND_RECOMMENDATION_NOTES.md`: 現行仕様なら維持、履歴ならアーカイブ。
- `implementation_plan.md`: 完了・未完了を照合後、必要部分を現行文書へ移し、追跡可能な形でアーカイブする。

削除を選ぶ場合の条件:

- 対応する情報が別文書に存在することを行単位で確認済み。
- README内リンクを更新済み。
- `rg -n "docs/|implementation_plan"` で壊れた参照がない。
- 削除理由をコミット本文に記録する。

### B2. `diva-data-pipeline` 側

- `ACTIVE/CURRENT_STATUS.md`: 2026-07-18時点の実装と実測テスト結果に更新。
- `ACTIVE/ROADMAP.md`: 完了項目は「完了済み」へ移し、コミットまたは検証根拠を添える。
- `ACTIVE/IMPLEMENTATION_PLAN.md`: 全項目の実装確認後に完了扱いとし、詳細ログは `ARCHIVE` へ移す。
- `ACTIVE/IDEAS.md`: チェック状態をコードと照合する。推測で完了にしない。
- `ARCHIVE/pdf_out.txt`: DIVA Playerとの関連がないことを確認後、独立コミットで削除する。

### B3. コミット分割

例:

1. `diva-player`: `docs: clarify public documentation ownership`
2. `diva-data-pipeline`: `docs: reconcile diva player active status`
3. `diva-data-pipeline`: `chore: remove unrelated archived OCR output`（削除する場合のみ）

文書だけの変更をSBCへデプロイする必要はない。実行物に変更がある場合だけ、通常のデプロイ手順を別途行う。

## 5. フェーズC: コード品質と文字コード確認

現在の基準はすでにgreenである。リファクタリングの目的を「警告を直す」から「回帰を防ぐ」に変更する。

1. `npm.cmd test`
2. `npm.cmd run lint`
3. `npm.cmd run build`
4. 文字化け候補を検索し、候補ごとに人手で判定する。

文字化けは単に非ASCII文字を探してはならない。日本語コメントは正常であるため、既知の化けパターンやUTF-8デコード失敗を対象にする。修正時はコメントだけの独立コミットとし、実行コードを同時変更しない。

追加テストの優先候補:

- 設定UIから各表示面までのグローバルフィルター統合テスト
- API候補をフロント側で除外した際の再取得上限と不足時表示
- DB migrationのfresh installと既存DB upgradeの両経路

## 6. フェーズD: DB未使用テーブルの監査

対象候補: `play_sessions`、`session_plays`。

### D1. 読み取り専用調査

- `diva-player` と `diva-data-pipeline` のコード、スクリプト、運用文書を検索。
- SBC上でテーブル行数、最終更新、依存ビュー、外部キー、権限を読み取り専用SQLで確認。
- APIアクセスログに旧エンドポイント利用がないか確認。
- fresh DBが `schema.sql`、既存DBが `migrations/*.sql` を通る構成差を確認。

### D2. 削除可否ゲート

次をすべて満たす場合だけDROPを提案する。

- コード参照、外部利用、運用利用がない。
- 保存すべきデータがない、またはバックアップ済み。
- ステージング相当環境でmigrationが成功。
- health checkだけでなく検索・推薦・履歴記録のスモークテストが成功。
- 復旧手順があり、所要時間を見積もっている。

### D3. 実装する場合

- 新規migrationで依存順にDROPする。
- fresh installで再生成されないよう `schema.sql` も同じ変更で更新する。
- migration適用記録がSQL本体と別トランザクションである現状を考慮し、途中失敗時の再実行性を確認する。
- 本番適用前にDBバックアップを取得し、復元テストまたは復元コマンドの確認を行う。

## 7. フェーズE: 未解決不具合

### E1. B3 非ボカロ楽曲の混入

まず調査タスクとして扱い、取り込み条件変更を同時に行わない。

1. 誤混入と判断された具体例を最低10件集める。
2. 正常曲を含む比較サンプルを用意する。
3. VocaDB属性、歌手種別、タグ、PV、既存品質フラグを比較する。
4. denylist、取り込み時除外、配信時除外、品質フラグの候補を、誤除外率と保守コストで比較する。
5. 選択したルールをfixture化し、誤混入例が除外され正常例が残る自動テストを追加する。
6. dry-runで影響件数を出し、承認後に適用する。

受け入れ条件:

- 報告された誤混入例が除外される。
- 比較用の正常曲が維持される。
- 除外理由を後から追跡できる。
- ルール解除またはdenylist復元の手順がある。

### E2. B4 関連曲・サジェストのボーカル偏重

保留を維持する。再開時は「同一ボーカル比率」「同一プロデューサー比率」「タグ多様性」を計測し、変更前後を比較する。感覚だけで重みを変更しない。

### E3. B5 音響情報欠損時のスコア補正

保留を維持する。再開時は欠損をボーナスにもペナルティにも暗黙変換せず、欠損フラグを説明可能にした上で、オフライン評価と推薦デバッグ出力で比較する。

## 8. フェーズF: ブランチ整理

機能・文書整備とは別の任意作業として最後に行う。

1. `git rev-list --left-right --count main...<branch>` でブランチ固有コミットが0であることを再確認。
2. 未コミット変更がないことを確認。
3. ローカルブランチを `git branch -d` で削除。
4. リモートブランチは、共有利用がないことを確認し、ユーザー承認後に削除。

`git stash` はブランチ削除に不要である。別ブランチに未コミット変更がある場合は、先に変更の所有者と扱いを確定する。

## 9. 検証マトリクス

| 変更 | 必須検証 |
|---|---|
| 文書整理 | 相対リンク確認、`rg`による旧参照確認、差分レビュー |
| TypeScript/React | `npm.cmd test`、`npm.cmd run lint`、`npm.cmd run build` |
| C# API | `dotnet test`または少なくとも`dotnet build`、対象APIスモークテスト |
| パイプライン | 対象スクリプトの単体テスト、dry-run、fixture比較 |
| DB migration | fresh install、既存DB upgrade、再実行性、バックアップ/復元確認 |
| SBC反映 | health、検索、推薦、履歴記録、ログ確認 |

## 10. 推奨実行順序

1. フェーズA: 未コミット変更の保護と文書対応表作成
2. フェーズB: 文書の正本・公開範囲を決定し、リポジトリ別に整理
3. フェーズC: green基準を維持しつつ不足する統合テストを追加
4. フェーズE1: B3を調査し、dry-run結果を提示
5. フェーズD: DBテーブルを読み取り専用で監査
6. 承認後のみ、B3ルール変更またはDB migrationを個別実施
7. 任意でフェーズFのブランチ整理

## 11. 実装前に必要な判断

実装開始前に必要なのは次の3点だけとする。

1. 公開リポジトリに残す文書範囲。推奨は、セットアップ・公開仕様・開発規約を `diva-player` に残し、SBC固有運用だけをprivate側に置くこと。
2. B3調査を今回のスコープに含めるか。
3. DBは読み取り調査までにするか、条件を満たした場合のmigration作成まで許可するか。本番適用は常に別承認とする。

`pdf_out.txt` の削除、過去計画のアーカイブ、リモートブランチ削除は独立した低優先度の保守判断であり、主要作業の開始条件にはしない。
