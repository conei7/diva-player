# 楽曲種別除外の確実化と次期機能 実装計画

## 0. この計画の扱い

- 本書は実装前の手順書であり、この段階ではソースコード、DB、SBC、Cloudflare Pages、GitHub には変更を加えない。
- 最優先は「除外する楽曲種別を選んでも対象曲が残るように見える」問題の再現、原因の切り分け、修正である。
- その後、現在の構成を流用して安全に追加できる機能を、依存関係順に実装する。
- ユーザーが明示的に保存したプレイリスト、履歴、高評価一覧からは、従来方針どおりグローバル除外によって曲自体を消さない。除外対象は検索、ホームの発見枠、おすすめ、関連曲、deep dig、自動キュー候補とする。

---

## 1. 現状調査で判明したこと

### 1.1 最も可能性が高い原因: 曲種を選んだだけでは有効にならない

現在の設定は次の二段階になっている。

1. `excludedSongTypes` に `Cover`、`Remix` などを登録する。
2. 別項目の `enabled`（「再生数・楽曲種別フィルターを有効にする」）をオンにする。

`src/utils/globalFilters.ts` の `matchesGlobalSongFilter` は `enabled === false` の場合、曲種が選択済みでも即座に `true` を返す。`src/stores/searchStore.ts` の `hasGlobalSongFilters` も同じ条件なので、バックエンド検索にも `excludeSongTypes` が送られない。

したがって、曲種チェック、全体有効化、最後の「適用」のいずれかが欠けると、設定値が画面に見えていても除外されない。この状態をユーザーが判別できる表示がないため、仕様上の分かりにくさが実質的な不具合になっている。

### 1.2 除外処理そのものは二系統に実装されている

- 通常検索は `searchStore.ts` から `/api/songs/search` に `excludeSongTypes` を渡し、SBC の `DbService.SearchSongsAsync` が `song_type NOT IN (...)` で除外する。
- ホーム、おすすめ、関連曲、deep dig、自動キューは取得後に `applyGlobalSongFilter` / `applyDiscoveryFilter` で除外する。

単体関数と検索APIの基本ロジックは存在するが、設定画面から各表示面までを通した統合テストが不足している。

### 1.3 「ユーザーがカバーだと思う曲」と `songType` が一致しない場合がある

現行判定は VocaDB / SBC の `song.songType` 完全一致だけを使用する。タイトル、タグ、歌手、`originalVersionId` から曲種を推測しない。そのため、メタデータが `Original` または `Unspecified` の曲は、内容がカバーに見えても `Cover` 除外では消えない。

この場合はフィルター漏れとメタデータ品質問題を区別する必要がある。タイトル文字列だけによる自動分類は誤除外が多いため、初回修正では採用しない。

### 1.4 適用面によってフィルター実行場所が異なる

通常検索はDB側で除外できる一方、急上昇、新着、推薦系API、Qdrant検索は候補取得後のフロントエンド除外が中心である。厳しい条件では、24件取得して大半を除外し、追加ページを最大8ページ先読みしても24件に届かない可能性がある。

## 2. 修正後の仕様

### 2.1 設定の有効・無効を曖昧にしない

推奨仕様は「条件を入力・選択した時点で曲フィルターを自動的に有効化する」である。

- YouTubeまたはニコニコの最低再生数を0より大きくする。
- 除外する楽曲種別を1件以上選ぶ。
- 上記操作を行った場合、ドラフトの `enabled` を自動的に `true` にする。
- フィルターを一時停止したい場合のため、全体トグルは残す。
- `enabled === false` のときは再生数・曲種入力欄を視覚的に無効化し、「条件は保存されていますが、現在は適用されていません」と表示する。
- 「適用」後、設定モーダル内とホーム上部に「適用中: Cover / Remixを除外、YouTube 10万以上」のような要約を表示する。
- 設定変更後に「適用」を押さず閉じようとした場合は、破棄確認を出すか、適用ボタン付近に「未適用の変更あり」を表示する。初回実装では後者を採用し、暗黙の自動保存は行わない。

### 2.2 曲種名を日本語化し、意味を明示する

内部値は既存APIとの互換性のため変更せず、UIラベルのみ次のようにする。

| 内部値 | 表示名 | 補足 |
|---|---|---|
| Original | オリジナル | 原曲として登録された曲 |
| Remaster | リマスター | 同一曲の再調整版 |
| Remix | リミックス | 原曲を再構成した版 |
| Cover | カバー | 他者・他音源によるカバー |
| Arrangement | アレンジ | 編曲版 |
| Instrumental | インスト | 歌唱なし版 |
| Mashup | マッシュアップ | 複数曲の混合 |
| MusicPV | 音楽PV | VocaDB上のMusicPV分類 |
| DramaPV | ドラマPV | 音楽以外を含む映像作品 |
| Other | その他 | その他分類 |
| Unspecified | 未分類 | 曲種が確定していないもの |

「未分類」を除外すると、メタデータ不足の曲をまとめて隠せる一方、正しいオリジナル曲も消える可能性がある旨を表示する。

### 2.3 除外理由を取得できる共通判定にする

`src/utils/globalFilters.ts` に、真偽値だけでなく理由を返す純粋関数を追加する。

```ts
type SongFilterRejectionReason =
  | 'disabled'
  | 'excluded-song-type'
  | 'youtube-views-missing'
  | 'youtube-views-below-minimum'
  | 'nico-views-missing'
  | 'nico-views-below-minimum'
  | 'rated-song'
  | 'cooldown';

type SongFilterDecision = {
  accepted: boolean;
  reason?: SongFilterRejectionReason;
};
```

- `matchesGlobalSongFilter` と `matchesDiscoveryFilter` はこの共通判定を利用する。
- 本番UIでは集計結果だけを表示し、曲ごとの詳細理由は開発用デバッグ表示に限定する。
- 「24件中 Cover 3件、Remix 2件、再生数不足4件を除外」のような件数を表示できるようにする。

### 2.4 適用対象を固定する

以下の表示面を必須対象とする。

- ホーム: あなたへのおすすめ、人気、急上昇、新着、マイナー発掘、最近聴いたPの曲
- 通常検索、P検索、ボーカリスト検索、詳細検索、追加読み込み
- 再生ページ: おすすめ、関連曲、deep dig
- 自動再生キューの候補補充
- 複数選択の「表示中の全曲」対象

以下には適用しない。

- 履歴
- 高評価一覧
- 手動プレイリスト
- 現在再生中の曲
- ユーザーがすでに手動でキューへ追加した曲

## 3. 実装手順

### Phase 0: 再現データと受け入れ基準を固定する

1. SBC検索APIから各 `songType` の代表曲を最低2曲ずつ取得する。
2. UI表示上もカードの曲種バッジとAPI値が一致することを確認する。
3. 次の組み合わせを再現表として保存する。

| enabled | 選択曲種 | 適用操作 | 期待値 |
|---|---|---|---|
| OFF | Cover | 適用 | 条件保存・適用停止の明示 |
| ON | Cover | 未適用で閉じる | 未適用の警告 |
| ON | Cover | 適用 | Coverが全対象面から消える |
| ON | Cover + Remix | 適用 | 両方が消える |
| ON | Unspecified | 適用 | 未分類が消え、注意表示が出る |

4. VocaDB上で内容と曲種が一致しない代表例があれば、フィルター不具合とは別の「メタデータ不一致」一覧に記録する。

### Phase 1: 設定状態とUIの修正

対象:

- `src/components/settings/SettingsModal.tsx`
- `src/stores/globalFilterStore.ts`
- `src/utils/globalFilters.ts`
- 新規 `src/components/settings/ActiveFilterSummary.tsx`（必要なら）

手順:

1. `hasConfiguredSongFilters(settings)` と `isGlobalSongFilterActive(settings)` を共通関数として定義する。
2. 再生数または曲種の変更時に、条件が存在すればドラフトの `enabled` をオンにする。
3. 全体トグルをオフにした場合は条件値を削除せず保持する。
4. オフ時は入力欄を `disabled` にし、保存済み条件が停止中であることを表示する。
5. 英語だけの曲種名を日本語ラベルと内部値の併記に変更する。
6. `draftFilters` と保存済み設定を比較してdirty状態を算出し、「未適用」を表示する。
7. 適用完了メッセージに実際の条件要約を含める。
8. ホームまたはヘッダー付近に、適用中条件と「解除」導線を持つ小さなフィルターチップを表示する。

### Phase 2: 曲種の正規化と共通判定

対象:

- `src/types/vocadb.ts`
- `src/utils/globalFilters.ts`
- `src/api/vocadb.ts`
- `src/stores/searchStore.ts`
- `backend/api/VocadbRecommender/Services/DbService.cs`

手順:

1. APIレスポンス読込時に `songType` が既知値か検証する。
2. 欠損・未知値は `Unspecified` として扱い、例外や無条件通過を防ぐ。
3. DBの `songs.song_type` と `raw_json->>'songType'` が不一致の件数を監査するSQLを用意する。
4. 不一致があれば同期パイプライン側で修正し、アプリ側で推測上書きしない。
5. `getSongFilterDecision` を導入し、既存の真偽値関数は互換ラッパーにする。
6. 全表示面が共通関数を経由しているか静的検索とテストで確認する。

### Phase 3: SBC側へのフィルター押し込み

目的は、候補を24件取得してから大量除外する挙動を減らし、厳しい条件でも可能な限り24件を満たすことである。

対象:

- `backend/api/VocadbRecommender/Program.cs`
- `backend/api/VocadbRecommender/Services/DbService.cs`
- `backend/api/VocadbRecommender/Services/RecommendService.cs`
- `backend/api/VocadbRecommender/Services/QdrantService.cs`
- `src/api/vocadb.ts`
- `src/pages/HomePage.tsx`
- `src/pages/NowPlayingPage.tsx`
- `src/pages/WatchPage.tsx`
- `src/hooks/useAutoQueue.ts`

手順:

1. 検索、trending、recommend、recommend/multi、metadata、audioで共有する `SongFilterRequest` を定義する。
2. `minYoutubeViews`、`minNicoViews`、`excludeSongTypes` を各APIへ渡す。
3. SQL候補はWHERE句で事前除外する。
4. Qdrant候補は必要件数より多く取得し、DBメタデータで除外後に件数を満たすまで追加取得する。
5. 推薦の多様性調整はフィルター後の候補に対して実行する。
6. フロントエンド側の最終フィルターは防御層として残す。
7. 追加ページの上限に達した場合は、無限にローディングせず「条件を満たす候補はN件でした」と表示する。

### Phase 4: 件数・空状態・診断表示

1. 検索結果件数はフィルター適用後の件数を表示する。バックエンド総件数と画面表示件数を混同しない。
2. 0件の場合は、現在の有効条件を列挙し、「曲種除外を解除」「再生数を0に戻す」「すべて初期化」の導線を表示する。
3. 24件未満の場合は、APIエラーと候補不足を別表示にする。
4. 開発用デバッグモードでは、取得数、除外理由別件数、採用数、追加取得ページ数を表示できるようにする。
5. 通常ユーザーには個別曲の除外ログを出さず、集計だけを表示する。

### Phase 5: テスト

#### フロントエンド単体テスト

- `enabled=false` では保存済み曲種が適用されないこと。ただしUIが停止状態を明示すること。
- `Cover`、`Remix`、複数曲種、`Unspecified` の除外。
- 未知の `songType` を `Unspecified` に正規化すること。
- 最低再生数と曲種除外を同時に適用した場合の優先理由。
- クールダウン、評価済み除外が曲種フィルターと併用できること。
- 設定の保存、再読込、旧バージョンからのmigration。

#### コンポーネントテスト

- 曲種選択で自動的に有効状態になる。
- トグルOFFで入力欄が無効になり、条件値は保持される。
- 未適用表示、適用後要約、初期化。
- 日本語ラベルと内部値の対応。

#### バックエンドテスト

- `excludeSongTypes=Cover` でCoverが0件になる。
- 複数値、未知値、空値、重複値。
- trending / recommend / Qdrant系でも除外後のレスポンスに対象曲種が含まれない。
- `start` / `offset` を使った追加取得で重複や取りこぼしがない。
- SQLパラメータ化が維持され、文字列連結による注入がない。

#### E2E

1. 設定でCoverとRemixを選択し適用する。
2. ホーム6カテゴリ、通常検索、再生ページ3タブ、自動キューを確認する。
3. 各画面で対象曲種が0件であることをカードのデータ属性または詳細情報で検証する。
4. リロード後も設定が維持されることを確認する。
5. 一時停止後は対象曲が再表示され、再有効化後は再び除外されることを確認する。
6. Cloudflare Pages経由とLAN/SBC直結の両方で実施する。

### Phase 6: 展開

1. 修正専用ブランチを作成する。
2. Phase 1〜2を1コミット、Phase 3をバックエンドとフロントエンドで分割、テストを別コミットにする。
3. 各コミットで `npm test`、lint、production build、C#テストを実行する。
4. SBCを先に更新し、healthと検索APIを確認する。
5. Cloudflare Pagesへ公開し、実URLでE2Eを行う。
6. GitHub Pagesのワークフローも確認する。
7. 問題があればコミット単位でrevertできるよう、機能追加とDB変更を混在させない。

## 4. 受け入れ条件

- 曲種を1件選んで適用した場合、全体トグルの状態が曖昧にならない。
- 適用中の除外条件を設定画面を開かなくても確認できる。
- `Cover` を除外した場合、`songType === 'Cover'` の曲が検索・発見・推薦・自動キューに1件も含まれない。
- 曲種メタデータが `Original` / `Unspecified` の曲を、タイトルだけで誤ってCoverとして除外しない。
- 厳しい条件でも追加候補を探索し、24件に届かない場合は候補不足を明示する。
- 履歴、高評価、プレイリスト、手動キューは消えない。
- ページ再読込、バックアップ、復元後も設定が壊れない。
- SBC経由の全対象API、Cloudflare Pages、GitHub Actionsが成功する。

## 5. 次に実装できる機能

IDEASの記載と現在のソースを照合すると、クールダウン、評価済み曲の発見候補除外、原曲へのジャンプ、アルバムのプレイリスト化はすでにコードが存在する。まずIDEASのチェック状態を実装実態に合わせて更新する。

### 優先度A: プロデューサー検索の曖昧一致修正

理由: 既存IDEASに具体的な不具合例があり、お気に入りP機能の前提にもなる。

1. `findArtistByName` の候補生成、完全一致、別名、読み、artist typeの優先順位をテスト化する。
2. 完全一致が複数ある場合、勝手に1件へ決めず小さな候補選択UIを表示する。
3. `Producer`、`Circle`、`Band` と歌声ライブラリを分離する。
4. 「MIMI」のような回帰ケースを固定する。
5. URLには選択済みartistIdを保持し、戻る・共有・再読込でも同じPを表示する。

### 優先度A: お気に入りP

依存: プロデューサー検索の安定化後。

1. `favoriteProducerStore` を作り、artistId、表示名、登録日時を保存する。
2. P名リンク、検索結果、楽曲詳細に登録・解除ボタンを追加する。
3. ホームに「お気に入りPの新着」カテゴリを追加する。
4. 推薦スコアには小さな加点だけを行い、同一P偏重を避ける上限を設ける。
5. 完全バックアップへ含める。
6. 削除済み・改名済みPの再解決処理を追加する。

### 優先度B: スマートプレイリスト

依存: 今回のフィルター判定共通化。

1. 条件モデルを `SmartPlaylistRule` として定義する。
2. 曲種、YouTube/Nico再生数、ボーカリスト、P、投稿年、評価、再生済みを組み合わせられるようにする。
3. 曲ID一覧は保存せず、開くたびに条件評価する。必要なら最終結果を短時間キャッシュする。
4. 通常プレイリストへのスナップショット変換を用意する。
5. 条件が解釈できない旧バックアップは無効項目として安全にスキップする。

### 優先度B: プレイリスト共有

1. 共有形式にschema versionを持たせる。
2. 少数曲は圧縮URL、大規模プレイリストはJSONファイルを使用する。
3. インポート前に曲数、名称、重複、無効曲をプレビューする。
4. 履歴や評価など不要な個人情報は共有データへ含めない。
5. URL長上限を超える場合は自動的にファイル共有へ誘導する。

### 優先度C: BPM検索

1. まずVocaDB APIとSBC DBにBPM値がどの程度存在するか充足率を調査する。
2. 充足率が十分なら詳細検索へ最小・最大BPMを追加する。
3. SBCに列がなければパイプライン、スキーマ、検索インデックスの順で追加する。
4. BPM不明曲を含めるか除外するかを明示的に選べるようにする。

### 後続候補

- YouTubeプレイリストの定期同期: OAuth、削除同期、並び順競合の設計が必要なため別計画にする。
- 知ってる度マップ: 集計定義と可視化方式を先に固める必要がある。
- 楽曲ベクトル宇宙表示: Qdrantベクトルの次元圧縮と大量描画の検証が必要で、独立プロジェクトとして扱う。
- 歌詞同期、クロスフェード、イコライザー: YouTube/Nicoの埋め込みプレイヤー制約を先に技術検証する。
